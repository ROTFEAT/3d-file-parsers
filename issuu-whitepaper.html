3d-tools.fabrapid.com Chapter 01 — The Problem

Chapter 01The Problem: Format Coverage Gaps

Three.js ships 30+ loaders. Assimp supports 70+ formats. Yet several real-world 3D formats remain completely unsupported in the JavaScript ecosystem.

Modern WebGL applications increasingly need to handle files from diverse sources — game engines, laser scanners, terrain generators, CAD tools, and scientific simulations. While the major formats (glTF, OBJ, STL, FBX, STEP) are well-covered by libraries like Three.js and Assimp compiled to WebAssembly, a long tail of niche formats has no JavaScript parser.

When we built 3d-tools.fabrapid.com — a browser-based 3D file converter supporting 70+ input formats — we hit this gap. Five specific formats required custom parsers written from scratch:

Format Origin Three.js Assimp.js Use Case
M3D (.m3d) Model3D project ✗ ✗ Lightweight game assets
X3D (.x3d) Web3D Consortium ✗ ✓ ISO standard 3D scenes
IrrMesh (.irrmesh) Irrlicht Engine ✗ ✗ Irrlicht game engine meshes
Terragen (.ter) Planetside Software ✗ ✗ Terrain heightmaps
PTS (.pts) Leica Geosystems ✗ ✗ Laser scan point clouds
Why Not Use Assimp WASM for Everything?

While Assimp's C++ library supports X3D, compiling the full Assimp to WebAssembly adds ~8MB to the bundle. For formats with simple, well-documented specifications, a purpose-built JavaScript parser is 100-1000x smaller and avoids WASM startup latency. Our X3D parser is 5KB vs Assimp's 8MB — a 1,600x size reduction for the same parsing task.

The Cost of Missing Format Support

Users of 3D conversion tools don't care which rendering engine you use internally. They have a file from Irrlicht, or a terrain from Terragen, or a point cloud from a Leica scanner — and they need it converted to glTF or OBJ. Every unsupported format is a user who leaves your application.

By open-sourcing these parsers, we aim to close the gap for the entire JavaScript 3D ecosystem — not just our own tool.

Chapter 02Format Deep Dive: 5 Unsupported Formats

Understanding the binary and XML structures behind each format is essential for writing correct, performant parsers.

M3D — Model3D Binary Format

M3D is a compact binary format designed as a modern replacement for older mesh formats. It uses a chunk-based structure similar to IFF/RIFF, with optional zlib compression.

File Structure

Binary Layout
┌──────────────────────────────────────────┐
│ Bytes 0-3   "3DMO" magic identifier      │
│ Bytes 4-7   File size (uint32 LE)        │
│ Byte  8     0x78 = zlib compressed        │
├──────────────────────────────────────────┤
│ HEAD chunk   scale + type flags           │
│ VRTS chunk   vertex data (int16/f32/f64)  │
│ MESH chunk   face indices + materials      │
│ OMD3         end-of-file marker           │
└──────────────────────────────────────────┘

The vertex precision is controlled by 2 bits in the HEAD flags field: 00 = int8, 01 = int16, 10 = float32, 11 = float64. All coordinates are scaled by the global scale factor from the HEAD chunk.

X3D — ISO Standard 3D Scenes

X3D is the XML-based successor to VRML97, standardized as ISO/IEC 19775. Despite being an official ISO standard, no pure JavaScript parser existed. Our parser handles the three most common geometry nodes:

  • IndexedFaceSet — arbitrary polygons with fan triangulation and -1 face separators
  • IndexedTriangleSet — pre-triangulated geometry with direct index mapping
  • Box primitive — parametric box generation with configurable dimensions

IrrMesh — Irrlicht Engine Meshes

The Irrlicht Engine uses an XML mesh format with three vertex types, each with different token strides per vertex:

Vertex TypeFieldsTokens per Vertex
standardposition + normal + color + UV9
2tcoordsstandard + second UV11
tangentsstandard + tangent + bitangent15

Terragen — Heightmap Terrains

Terragen .ter files store elevation data as a grid of 16-bit integers with scale metadata. Our parser automatically triangulates the heightmap into a renderable mesh (2 triangles per grid quad) and computes smooth vertex normals.

PTS — Leica Point Clouds

PTS is a simple ASCII format from Leica laser scanners. Each line contains XYZ coordinates with optional intensity and RGB color. Files can contain millions of points, so parsing performance is critical.

Chapter 03Architecture: Framework-Agnostic Design

Every parser returns raw Float32Array/Uint32Array data — no Three.js, no Babylon.js, no framework lock-in.

Design Decision: Raw Typed Arrays

Most 3D file loaders in the JavaScript ecosystem return framework-specific objects. Three.js loaders return BufferGeometry. Babylon.js plugins return Mesh objects. This creates an unnecessary coupling between file parsing (a pure data operation) and rendering (a framework-specific concern).

Our parsers separate these concerns completely:

Output Contract

Every mesh parser returns the same core fields:

FieldTypeDescription
positionsFloat32ArrayFlat XYZ vertex positions (length = vertexCount × 3)
normalsFloat32Array | nullPer-vertex normals (auto-computed if not in file)
indicesUint32ArrayTriangle indices (length = faceCount × 3)

This output maps directly to WebGL buffer data — no conversion needed. You can pass these arrays straight into gl.bufferData(), THREE.BufferAttribute(), or BABYLON.VertexData.

Dependency Strategy

Zero-Dep Parsers (4 of 5)

X3D, IrrMesh, Terragen, and PTS parsers have zero npm dependencies. They use only built-in browser APIs (DataView, TextDecoder, DOMParser).

Single Dependency: fflate

The M3D parser requires fflate (3KB gzipped) for zlib decompression. We chose fflate over pako for its smaller size and better tree-shaking support.

Tree-Shaking Support

Each parser is independently importable. If you only need the Terragen parser, your bundle only includes that parser's code:

TypeScript
// Only 4KB in your bundle — not the full 33KB package
import { parseTerragen } from '3d-file-parsers/terragen';

Chapter 04Implementation: Code Walkthrough

Key implementation details and techniques used in each parser.

Binary Parsing with DataView

The M3D and Terragen parsers process raw binary data using JavaScript's DataView API, which handles endianness correctly and provides typed access to buffer contents:

TypeScript
const view = new DataView(buffer);

// Read little-endian float32 at byte offset
const x = view.getFloat32(offset, true);

// Read little-endian int16 (Terragen elevation)
const elevation = view.getInt16(offset, true);

Polygon Fan Triangulation

Both M3D and X3D support arbitrary polygons (quads, n-gons). We convert these to triangles using fan triangulation — a simple O(n) algorithm that works for convex polygons:

TypeScript
// Fan triangulation: vertex 0 connects to all edges
// Polygon [A, B, C, D, E] → triangles [A,B,C], [A,C,D], [A,D,E]
for (let i = 1; i < polygon.length - 1; i++) {
  triangles.push(polygon[0], polygon[i], polygon[i + 1]);
}

Automatic Normal Computation

When files don't include vertex normals (M3D, X3D, Terragen), we compute smooth normals by accumulating face normals at each vertex, then normalizing:

TypeScript
// For each triangle: compute face normal via cross product
const e1 = [bx - ax, by - ay, bz - az]; // edge 1
const e2 = [cx - ax, cy - ay, cz - az]; // edge 2
const normal = cross(e1, e2);

// Accumulate at each vertex → average gives smooth normals
for (const idx of [a, b, c]) {
  normals[idx * 3]     += normal[0];
  normals[idx * 3 + 1] += normal[1];
  normals[idx * 3 + 2] += normal[2];
}

// Normalize all vertex normals to unit length
const len = Math.sqrt(x*x + y*y + z*z) || 1;
normals[i] = x / len;

Isomorphic XML Parsing

X3D and IrrMesh are XML-based formats. Our parsers work in both browsers and Node.js by accepting an optional DOMParser parameter:

TypeScript
// Browser: uses built-in DOMParser automatically
const data = parseX3D(xmlText);

// Node.js: pass @xmldom/xmldom
import { DOMParser } from '@xmldom/xmldom';
const data = parseX3D(xmlText, new DOMParser());

Zlib Decompression for M3D

M3D files may be zlib-compressed (detected by byte 8 being 0x78). We use the fflate library for decompression — it's 3KB gzipped and 2-3x faster than pako:

TypeScript
import { unzlibSync } from 'fflate';

let chunkData: Uint8Array;
if (fileData[8] === 0x78) {
  chunkData = unzlibSync(fileData.slice(8)); // decompress
} else {
  chunkData = fileData.slice(8);            // already raw
}

Chapter 05Integration with Three.js & Babylon.js

Bridging raw typed arrays to your rendering framework takes just a few lines of code.

Three.js Integration

TypeScript
import * as THREE from 'three';
import { parseM3D } from '3d-file-parsers';

const data = parseM3D(buffer);
const geometry = new THREE.BufferGeometry();

geometry.setAttribute('position',
  new THREE.Float32BufferAttribute(data.positions, 3));

if (data.normals) {
  geometry.setAttribute('normal',
    new THREE.Float32BufferAttribute(data.normals, 3));
}

geometry.setIndex(
  new THREE.BufferAttribute(data.indices, 1));

const mesh = new THREE.Mesh(geometry,
  new THREE.MeshStandardMaterial());
scene.add(mesh);

Babylon.js Integration

TypeScript
import * as BABYLON from '@babylonjs/core';
import { parseTerragen } from '3d-file-parsers';

const data = parseTerragen(buffer);
const mesh = new BABYLON.Mesh('terrain', scene);
const vertexData = new BABYLON.VertexData();

vertexData.positions = Array.from(data.positions);
vertexData.indices   = Array.from(data.indices);
vertexData.normals   = Array.from(data.normals);

vertexData.applyToMesh(mesh);

Point Cloud Rendering with Three.js

TypeScript
import { parsePTS } from '3d-file-parsers';

const data = parsePTS(buffer);
const geometry = new THREE.BufferGeometry();

geometry.setAttribute('position',
  new THREE.Float32BufferAttribute(data.positions, 3));

if (data.colors) {
  geometry.setAttribute('color',
    new THREE.Float32BufferAttribute(data.colors, 3));
}

const points = new THREE.Points(geometry,
  new THREE.PointsMaterial({
    size: 0.01, vertexColors: data.hasColor
  }));
scene.add(points);
Zero-Copy Performance

Because our parsers output the exact same Float32Array format that WebGL expects, there's no data transformation overhead. The typed arrays go directly from the parser into GPU buffer memory via gl.bufferData() — the same path that framework-native loaders use internally.

Web Worker Compatibility

All parsers work inside Web Workers for non-blocking parsing. At 3d-tools.fabrapid.com, we run these parsers in dedicated workers to keep the UI thread responsive during large file processing.
